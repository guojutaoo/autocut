\---  # 视频生成自验证与迭代修复 Skill  ## 1. 目的与适用场景  为所有“基于指令自动生成视频/图片/音频”的任务提供统一的质量护栏：  - 每次生成任务必须经过“生成 → 自动验证 → 不合格则迭代修复”的闭环； - 不允许在未经过验证或验证失败的状态下，直接将结果交付给用户； - 支持的产物类型包括但不限于：短视频、封面图、插画、配乐/配音等多媒体内容。  典型使用场景： - 根据文案自动生成一个讲解视频，并确保视频时长、节奏、字幕位置等符合预设标准； - 自动生成包含特定文案与构图的封面图片，并确保分辨率与安全边距符合投放规范； - 自动生成一段背景音乐或配音，并确保时长与响度在安全区间内。  ## 2. 核心原则  - 始终执行“先生成、再验证、再修复”的循环流程； - 将验收标准显式化，写入配置，而不是隐含在 Prompt 里； - 尽量用可度量的指标来判断结果是否达标（分辨率、时长、帧率、响度等），对“创意是否好看”类主观问题保持保守； - 对于每一次验证失败，都要给出清晰的失败原因和下一轮调整方案； - 控制最大迭代次数，避免无限循环。  ## 3. 输入约定  在执行本 Skill 前，需要具备以下输入信息（可以来自用户说明或配置文件）：  1. \*\*需求说明\*\*（自然语言）：    - 要生成的内容类型：视频 / 图片 / 音频；    - 简要内容：例如“介绍 XXX 的 30 秒竖版视频，带中文字幕和轻快 BGM”；    - 重要约束：例如“必须是 9:16”、“结尾停留 1 秒”、“字幕不遮挡人物脸”。  2. \*\*生成配置（建议 JSON/YAML 形式）\*\*：    - \`type\`: \`video\` / \`image\` / \`audio\` / \`mixed\`；    - \`generator\`: 指定要调用的生成方式或工具（例如某个内部视频生成 API、脚本管线、ffmpeg 命令模板等）；    - \`params\`: 初始参数（如分辨率、时长、定格时间、字幕样式、BGM 音量等）；    - \`acceptance\_criteria\`: 验收标准（按类型拆分，见下一节）；    - \`max\_iterations\`: 最大迭代次数（例如 3\~5 次）；    - \`fallback\_strategy\`: 在多次失败后的回退策略（例如“退回人工审核”、“只输出问题报告不交付结果”）。  3. \*\*生成结果的输出路径约定\*\*：    - 需要约定好生成脚本/工具会把视频、图片、音频产物输出到什么路径，以便后续验证阶段读取。  ## 4. 验收标准示例  ### 4.1 视频验收标准  在配置中为 \`video\` 类任务定义如下字段（可按需要扩展）：  - 基础属性：   - \`width\` / \`height\`: 目标分辨率（例如 1080x1920）；   - \`fps\`: 目标帧率区间（例如 24\~30 fps）；   - \`duration\`: 目标时长区间（例如 28\~32 秒）；  - 画面与节奏：   - \`freeze.max\_segments\`: 允许的“异常静帧”段数量上限；   - \`freeze.min\_duration\`: 判定静帧段的最小时长（例如 >= 0.5 秒）；   - \`blackframe.max\_ratio\`: 黑帧占比上限（如 < 3%）；   - 关键时间点约束：例如 \`beats: \[0.0, 5.0, 10.0]\` 表示在这些秒数附近应该出现节奏点或画面变化，可用简单的画面差分/音频能量变化做粗检；  - 文案与字幕（可选）：   - 字幕是否存在：例如必须在 X\~Y 秒区间出现指定文本；   - 字幕区域约束：例如字幕底边到画面底部的距离在 80\~160 像素；  - 音频部分（若视频自带音频）：   - 允许的整体响度区间；   - 是否存在长时间静音段；  ### 4.2 图片验收标准  - \`width\` / \`height\`: 输出尺寸； - \`format\`: 图片格式（如 PNG/JPEG）； - \`alpha\`: 是否需要透明通道； - \`margins\`: 主要元素距离边缘的最小安全边距； - 关键元素存在性：例如必须包含某段文案或 Logo（可以通过 OCR / 简单视觉分析工具粗检）。  ### 4.3 音频验收标准  - \`duration\`: 时长区间（例如 与目标视频时长差不超过 0.5 秒）； - \`sample\_rate\`: 采样率要求； - 响度与峰值：   - 目标整体响度区间（例如 -23 \~ -16 LUFS）；   - 峰值不超过某阈值（避免削波）； - 静音段：不允许长时间静音（除非有明确需求）。  ## 5. 执行流程  执行本 Skill 时，遵循如下总体流程：  1. 解析需求说明与生成配置，构建内部任务描述与验收规则； 2. 初始化迭代计数 \`iteration = 1\`； 3. 进入循环：    1. 根据当前配置执行一次“生成步骤”：       - 调用约定的生成工具（例如视频生成脚本、图片生成模型、音频合成器等）；       - 等待生成完成，记录产物路径；    2. 执行“验证步骤”：       - 根据产物类型（视频/图片/音频）选择对应的验证流程；       - 读取产物的基础属性（尺寸、时长、帧率、响度等）；       - 调用多模态分析工具（例如对视频抽帧、OCR 文本、ASR 语音转写等），对照验收标准；       - 汇总所有检查项，生成结构化的验证结果（例如 JSON 对象，包含各项是否通过及说明）；    3. 根据验证结果判断：       - 若所有必选验收项均通过：         - 记录“验收通过”的结论，退出循环，将产物作为最终结果；       - 若存在不通过项：         - 根据具体失败项，构造“修复计划”：           - 技术参数问题（分辨率/时长/帧率/响度等） → 优先通过调整生成参数或简单滤镜修正；           - 节奏/定格/字幕位置等问题 → 调整相关的时间点、偏移量或重新生成对应片段；           - 内容严重偏离需求 → 考虑调整 Prompt 或重新生成；         - 记录当前迭代的失败原因与修复计划；    4. 检查迭代次数：       - 若 \`iteration >= max\_iterations\` 且仍未通过：         - 执行回退策略（例如输出“多轮尝试仍未达标”的问题报告，不交付产物，或提示转人工审核）；         - 退出循环；       - 否则：         - 根据修复计划更新生成配置（例如调整参数、替换部分片段或文案）；         - \`iteration += 1\`，返回步骤 3.1 继续下一轮。  ## 6. 验证与自我检查策略  在“验证步骤”中，建议按以下层次进行自我检查：  1. \*\*快速结构检查\*\*：    - 文件是否存在且可读；    - 基础元数据是否符合预期（视频/图片/音频类型是否正确，编码是否可解码等）。  2. \*\*硬性指标检查\*\*（必须通过）：    - 分辨率、帧率、时长、格式、采样率等硬性条件；    - 黑帧/静帧/静音段是否超过阈值；  3. \*\*软性指标检查\*\*（尽量满足）：    - 节奏点大致是否契合预期时间点；    - 字幕/文案大致是否在预期位置和时间范围内出现；  4. \*\*多模态内容对齐\*\*：    - 对视频：      - 抽取关键帧，进行简单的图像内容分析（颜色、构图变化）和 OCR；      - 对照用户需求中提到的关键元素（如“出现某个产品”、“出现某句标语”）。    - 对音频：      - 使用语音识别获取转写文本，对照脚本；      - 检查音频能量随时间的变化，粗略对齐节奏点。  所有检查过程需要输出清晰的结构化结果，供后续迭代决策使用，例如：  \`\`\`json {   "passed": false,   "issues": \[     {"type": "duration", "message": "实际时长 25.1s，低于下限 28s"},     {"type": "subtitle\_margin", "message": "字幕距底边仅 20px，小于最小安全边距 80px"}   ],   "suggested\_fixes": \[     {"param": "duration", "action": "increase", "value": 30.0},     {"param": "subtitle\_margin", "action": "increase", "value": 100}   ] }
