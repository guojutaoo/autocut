# LLM 投喂信息盘点（基于 transcript\_for\_llm.txt）

## 目标

盘点“当前代码已经产出/可直接给到 LLM 的信息”，并给出一套**最小但信息密度高**的投喂材料包组织方式，帮助 LLM 更稳定地：

* 选段（起承转合、桥接段、避免跳跃）

* 选策略（freeze/overdub/pure\_audio/slow\_mo）

* 写旁白（更贴画面、更贴节奏）

* 控制风险（ASR 错字、台词密集禁 overdub、结尾必须覆盖 80%-100%）

> 约束：这里仅列“目前代码已有的信息/文件”，不要求新增算法或改架构。

***

## 现有材料来源（按产物文件）

### 1) transcript\_for\_llm.txt（逐句台词 + 多模态标签）

文件路径：`outputs/autocut_project/transcript_for_llm.txt`

**每行结构（当前实际写法）**：

* 时间范围：`[start - end]`（格式为 `HH:MM:SS,mmm`）

* 台词：`台词：“...”`

* 标签：`特征：【视觉：...】 | 【听觉：...】 | 【节奏：...】 | 【台词密度：...】`

**对 LLM 的价值（可直接用）**：

* **剧情理解**：逐句台词 + 精确时间戳（可定位回原片）

* **策略选择**：

  * 台词密度：密集/稀疏/无台词（决定 freeze vs pure\_audio vs overdub）

  * 听觉：死寂/嘈杂（决定是否需要旁白补强、是否需要降噪/ducking）

  * 视觉：剧烈晃动/静态（决定 slow\_mo / freeze 是否合适）

  * 节奏：语速快/慢（决定旁白句长、字幕切分速度）

* **桥接段识别**：当剧情跳跃大时，可用“动机改变/新人物出场/环境变迁”提示从中间补桥段

**这些标签在代码中的来源/规则（便于你评估可靠性）**：

* 视觉强度：灰度 std 的 max（阈值离散化为 3 档）

* 听觉能量：RMS max（阈值离散化为 3 档）

* 台词密度：台词覆盖率（窗口为 start/end 左右各扩 2s）

* 节奏：字幕字符数 / 时长（cps）

***

### 2) compose\_plan.json（预选片段计划 / LLM 参考）

文件路径：`outputs/autocut_project/compose_plan.json`

**LLM 可用字段（预选 plan 的“轻量版”）**：

* `source_video`：源视频路径

* `full_transcript_path`：`transcript_for_llm.txt` 路径

* `segments[]`：候选片段数组，每段包含：

  * `start/end`：候选片段时间范围（秒）

  * `score`：系统选段打分

  * `selection_reason`：选段原因（面向人类/LLM 的解释）

  * `subtitle_texts`：该段包含的字幕文本数组（便于 LLM 快速读段内剧情）

  * `narration_text`：模板占位解说（可被 LLM 重写/删改）

**对 LLM 的价值（可直接用）**：

* **降噪**：把整集几万字台词压缩成“优先看”的候选段落

* **可解释性**：selection\_reason 给 LLM 一个“为什么这段可能精彩”的先验

* **段内台词摘要**：subtitle\_texts 相当于“这段发生了什么”的快速摘要

* **段级结构**：start/end 用于 LLM 输出“严格可渲染”的 JSON 剧本

***

### 3) video\_script\_expert\_output.json（LLM 最终剧本输出）

文件路径：`outputs/autocut_project/video_script_expert_output.json`

**对 LLM 的价值（作为二次自检材料）**：

* 把它当作“草稿 -> 自审 -> 再输出”的输入，可以要求 LLM：

  * 检查是否覆盖 80%-100% 的结尾红线

  * 检查段间跳跃是否需要桥接段

  * 检查策略配比（避免全 freeze 审美疲劳）

  * 检查 `pure_audio` 是否保证 narration\_text 为空

## 现有“隐含信息”（不新增算法，但可以用现有字段推断）

### A) 段间跳跃强度

来自：`compose_plan.json` 或 LLM 输出 segments 的时间轴

* 计算相邻段 `gap = next.start - prev.end`

* **用途**：当 gap > 10min 时，强制提醒 LLM 做桥接段检查

### B) 角色/场景变化迹象（弱信号）

来自：`transcript_for_llm.txt` 的台词内容本身 + 多模态标签

* 台词主体变化（称谓、地名、人名高频词变化）

* 视觉从“静态/阴暗”变成“剧烈晃动/高能”

* 听觉从“死寂”变成“能量爆发”

* **用途**：辅助判断“转场/转折点”是否成立

### C) 旁白可读性风险

来自：`节奏/台词密度`

* 台词密集 + 语速极快：旁白不宜长句，字幕更需要短句切分

* **用途**：指导 LLM 控制每段旁白字数、避免拥挤

## LLM 需要“视觉内容”吗？（结论与取舍）

### 结论（分层答案）
- **不强依赖**：如果目标是“按剧情解说剪辑”，LLM 仅靠台词 + 时间戳 + 候选段（compose_plan）就能产出可用剧本；视觉内容不是硬依赖。
- **强增益**：当你希望它做到“更贴画面”的解说（人物在做什么、动作细节、场面调性），或更准地选 `render_type`（slow_mo/overdub/pure_audio），视觉信息能显著减少胡编与误判。

### 当前代码已经能给的“视觉代理信号”（不传图片也有用）
- `特征：【视觉：静态/平稳 | 静态/阴暗 | 剧烈晃动/高能】`：指导 LLM 选择策略与措辞风格（冷静拆解/情绪爆点/慢动作细节）。
- `特征：【听觉】+【台词密度】+【节奏】`：与视觉标签组合，可推断“打斗/争吵/静默对峙/对白推进”等类型。
- `selection_reason`：候选段的可解释线索（间接包含画面强度/情绪点的先验）。

### 如果要给“真正的视觉内容”，最小可行形态
为了不把上下文炸掉，推荐从轻到重三档：
1. **每段 1 张关键帧（推荐）**：对每个候选 segment 在 `anchor_time` 或 `mid=(start+end)/2` 抽 1 张图，作为“画面锚点”。
2. **每段 2-3 张关键帧**：start/anchor/end 各一张，帮助理解“这段发生了什么变化”。
3. **每段文字化视觉描述（不推荐先做）**：需要额外模型/规则生成描述，文本 token 成本高且容易引入噪声。

### 建议落地策略（与你当前流程匹配）
- 默认只投喂“视觉代理信号”（四类标签 + 候选段台词摘要），先把“主线结构/策略配比/结尾红线”稳定住。
- 当出现“解说不贴画面/选段误判动作场面”时，再升级到“每段 1 张关键帧”。

## 基于 OpenCV 能捕捉“都有谁/什么动作表情”吗？

### 先说结论（避免误解）
- **OpenCV 本身不是“识人/识动作/识表情”的模型**，它更像工具箱（读帧、跟踪、几何/光流、DNN 推理入口）。  
- 要做到“画面都有谁、在做什么动作、什么表情”，通常需要**额外的已训练模型**（可以通过 OpenCV DNN/ONNX 跑，也可以用别的库）。  
- 在“短视频解说剪辑”这个业务里，**不需要到‘识别角色身份’这一级**也能给 LLM 提供很大增益：比如“几个人、是否近景脸、是否激烈运动、是否有人在说话/对峙”等。

### 在不引入复杂训练的前提下（OpenCV + 现成模型）可做到的“高性价比”视觉特征
下面每项都可以最终变成给 LLM 的结构化字段（不必传全帧）：

1. **人/脸的存在与数量（Who 的替代指标）**
   - 能做：人物数量、脸数量、每张脸的 bbox、画面中脸的占比（近景/远景）。
   - 不能做：稳定地“这是谁（角色名/演员）”——除非做专门的人脸识别库 + 角色库对齐。

2. **说话/对话强度（谁在说话的弱推断）**
   - 能做：用“嘴部区域运动量 / 口型开合（landmark）”粗略判断“可能在说话的脸”。
   - 适合：辅助决定 `pure_audio`（原声对话）/ `freeze`（拆解台词）/ `overdub`（补旁白）。

3. **动作强度/镜头运动（What happens 的粗刻画）**
   - 能做：光流强度、人物 bbox 速度、镜头抖动（全局运动），区分“打斗/奔跑/追逐/静坐对峙”。
   - 适合：辅助选择 slow_mo、以及旁白用词（“冲出/怒吼/沉默对视” vs “缓缓抬眼/低声”）。

4. **姿态（Pose）与简单动作类别（What action）**
   - 能做：引入现成 pose 模型（如 MediaPipe Pose / OpenPose / YOLO-Pose），得到关键点；再用规则/轻量分类器归纳动作（站/坐/跪/拥抱/挥手/指向）。
   - 注意：这一步已经超出“纯 OpenCV”，但实现仍可以是“下载模型 -> 推理 -> 产出 keypoints”。

5. **表情（Expression）**
   - 能做：基于人脸 landmark + 轻量情绪分类器（smile/angry/neutral 等）做**粗粒度**判断。
   - 不能做：复杂微表情（厌恶/轻蔑/隐忍）稳定识别，误差会比较大。

### 建议的落地顺序（从稳到难）
1. **只做“人脸/人数/近景程度 + 运动强度”**（最稳，几乎不引入幻觉）
2. 再加 **“谁在说话的弱推断（嘴部运动）”**
3. 最后再考虑 **pose/表情**（需要额外模型，且要做置信度与降噪）

## 投喂材料包建议（最小集合）

为了控制上下文长度，同时让 LLM 做出稳定决策，建议每次投喂只包含：

1. **头部摘要**（你在提示词里写，不靠文件）：

   * 视频总时长（可由 transcript 最后一行时间得到）

   * 输出目标时长（5-7 分钟等）

   * 强制红线（覆盖 80%-100%、策略配比、台词密集禁 overdub…）
2. **compose\_plan.json 的 segments 列表**（候选段 + reason + subtitle\_texts）
3. **transcript\_for\_llm.txt 的“抽样视窗”**：

   * 只提供：每个候选段 start/end 前后各 N 秒内的台词行（而不是全文件 800+ 行）

   * 结尾再附上最后 30 行（用于结尾红线）

这样能把 LLM 上下文从“全片台词”压缩到“候选段上下文 + 结尾锚点”，显著降低 token。
